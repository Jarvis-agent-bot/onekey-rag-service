# code is cheap，show me the proto

## 核心理念
我们相信“少争论，多交付”。  
与其在会议里争对错，不如先把能看见的价值做出来。  
再慢慢补齐那些“不可见的工程”。  

有人说“Talk is cheap, show me the code”。  
这句话出自 Linux 之父林纳斯·托瓦兹。  
“Talk is cheap, show me the code”（空谈廉价，给我看代码）这句名言出自Linux之父林纳斯·托瓦兹（Linus Torvalds）
我们想说：**talk 当然也很重要，但原型更诚实**。  

## 我们的开发范式：文档驱动 + 原型驱动
文档是为了对齐边界，原型是为了校验体验。  
两者合起来，才能又快又稳。

## 标准流程（从对齐到交付）

### 第 0 步：需求启动与边界确认
先停一停，问清楚再出发。  
我们只做值得做的事，能说清“为什么做”再开始。
- 输入：一句话需求、背景、干系人、时间节点
- 输出：需求结论（做/不做/延后）+ 核心假设清单

### 第 1 步：需求文档生成（Skill1）
文档不是官样文章，而是“团队共同理解的一张地图”。  
它不求厚，只求清楚、可执行、可验收。
- 输入：背景、目标用户、业务目标、约束与风险
- 输出：
  - 目标与不做清单
  - 目标用户与关键场景
  - 成功标准与验收口径
  - 范围边界与关键流程
  - 非功能要求与依赖风险

### 第 2 步：Stitch 原型验证（所见即所得）
眼见为实，原型是最快的沟通方式。  
让 UI 自己说话，让交互自然纠错。
- 输入：文档里的“产品一句话 + 主要页面 + 关键交互”
- 输出：原型稿（页面结构、导航关系、关键交互路径）

### 第 3 步：原型细化输出（Skill2）
把“看起来对”的东西，变成“做起来不迷路”。  
原型是答案的雏形，细化是交付的骨架。
- 输入：Stitch 导出的 HTML/图片 + 需求文档 V1
- 输出：
  - 页面清单与页面目的
  - 字段规则与校验逻辑
  - 状态流转与异常处理
  - 接口草案与数据假设
  - 页面级验收清单

### 第 4 步：工程开发与验收闭环
开发就是把“说清楚的事”做出来，再让数据告诉我们结果。  
交付不是终点，闭环才是。
- 输入：需求文档 V1 + 原型稿 + 页面级规格
- 输出：可交付版本 + 验收记录 + 需求变更日志

## 产出物清单（按阶段）
- 需求启动：需求结论、假设清单
- 文档阶段：需求文档 V1
- 原型阶段：原型稿与交互路径
- 细化阶段：页面级规格文档
- 交付阶段：验收记录与变更日志

## 服务的“技能化”方向
我们不只做功能，还把能力做成可复用的流程技能。  
比如需求文档生成、原型细化、性能优化、代码审查、UI/UX 优化、测试自动化。  
让系统具备持续进化的能力，而不是一次性交付。

## 结语
少讲概念，多给原型；少做猜测，多做验证；少堆代码，多做闭环。  
code is cheap，show me the proto。
